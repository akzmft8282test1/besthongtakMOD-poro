01. 서론 및 보안 아키텍처 개요

본 기술 문서는 Phaser 3 게임 엔진으로 컴파일된 브라우저 게임 PokeRogue의 프론트엔드 환경에서 구동되는 확장형 대시보드 커널 모듈(mod.min.js)의 아키텍처 및 통신 보안 매커니즘을 상세히 다룹니다. 본 모듈의 핵심 설계 사상은 기존 클라이언트 주입 방식의 비보안성 치트 프로그램의 취약점을 보완하고, 무단 배포와 변조를 방지하기 위해 정교한 백엔드 검증 레이어를 결합하여 설계된 점입니다.

클라이언트 브라우저 런타임 가동과 동시에, 스크립트는 자체 메모리 컨텍스트 내에 익명 샌드박스를 구성하여 외부 디버거(DevTools)의 후킹 분석을 우회합니다. 또한, 내부 게임 메모리의 스택 필드나 힙 데이터를 단순 오버라이딩하는 것을 넘어 게임의 데이터 동기화 파이프라인(SaveSystem) 및 UI 렌더링 핸들러 스택에 직접 가입하여 실시간 데이터 일관성을 유지하는 복합적인 알고리즘이 내장되어 있습니다.

💡 핵심 설계 철학

게임 서버와 클라이언트 간의 통신 부하를 최소화하면서도, 로컬 클라이언트 단의 주요 가치 변조 내역이 유실되거나 인게임 저장 시스템에 의해 거부되지 않도록 설계 단계부터 트랜잭션 동기화 모듈을 전면 배치하였습니다.

02. 다단계 사용자 검증 흐름 및 15자리 UID 매커니즘

전통적인 일회성 시리얼 키 코드의 인증 방식은 토큰의 탈취, 중복 사용, 그리고 복제 사용에 매우 취약합니다. 본 엔진 대시보드는 이러한 보안 결함을 방어하고자 다단계 사용자 식별 검증 흐름(Multi-Stage Verification Process)을 채택했습니다. 사용자는 확장 프로그램의 컨텍스트 팝업 창에서 발급 코드를 최초 입력하게 되며, 이는 중앙 제어 서버에서 엄격히 통제되는 난수 테이블과 대조를 이룹니다.

이 과정에서 생성되는 15자리의 식별 문자열인 ROGUE-UID[XXXXXX] 구조는 다음과 같은 규격으로 파싱 및 관리됩니다.

[ Chrome Ext 팝업 창 ]
쿠폰 입력 요청
[ Render 중앙 검증 서버 ]
│
▼ (쿠폰 고유성 확인 및 즉시 소멸 처리)
[ 15자리 유저 고유 UID 생성 ]
◀── 클립보드 자동 동기화 복사
[ ROGUE-UID[XXXXXX] 포맷 ]
│
▼
[ 인게임 대시보드 패널 ]
UID 복사본 입력 제출
[ 세션 해금 및 기능 활성화 ]

UID 내부 구조 포맷 세부 사항

생성되는 유저 UID는 반드시 15자리의 엄격한 정적 텍스트 길이를 가집니다. 앞선 10글자의 고정 프리픽스인 ROGUE-UID[와 뒤이은 대문자 알파벳 및 0-9 사이의 숫자로 난수 생성된 6자리 문자 조합 중 일부를 마스킹하여 구성한 뒤 최종 브래킷 문자(])로 닫아 정확히 15바이트의 스트링 데이터를 충족시킵니다.

  • 쿠폰 즉시 소멸 속성(Burn-On-Redeem): 사용자가 쿠폰을 통해 한 번 UID를 발급받으면 해당 쿠폰은 DB 내에서 is_used = true 상태로 플래그가 전전되어 재사용이 원천 봉쇄됩니다.
  • 클립보드 가로채기 차단: 팝업 스크립트의 navigator.clipboard.writeText API를 통해 발급 성공 즉시 사용자 OS 클립보드 버퍼에 자동 안착하여 메모리 단 유출 위험을 줄였습니다.

03. 백엔드 라이선스 세션 인터페이스 분리 명세

중앙 통제 서버인 NodeJS 익스프레스 애플리케이션과의 원격 통신은 완전한 REST API 인터페이스 명세를 준수합니다. 엔드포인트는 쿠폰을 고유 식별자로 교환하는 교환기(/api/license/exchange-uid)와 인게임 대시보드 내부 진입 시 실시간 라이선스를 검증하는 검증기(/api/license/verify-uid)로 구조가 완벽히 분리되어 동작합니다.

REST 엔드포인트 통신 및 데이터 구조 매트릭스

엔드포인트 URI HTTP 메서드 요청 페이로드 데이터 응답 데이터 명세 보안 수준 및 역할
/api/license/exchange-uid POST { "couponCode": "STR" } { "valid": true, "userUid": "STR" } High 쿠폰 1회 소멸 및 고유 15자리 UID 발급
/api/license/verify-uid POST { "userUid": "STR" } { "valid": true, "token": "JWT", "features": [], "durationMinutes": 0 } Critical 런타임 패널 권한 배열 교부 및 만료 시간 셋업
/api/admin/generate-coupon POST { "customCode": "STR", "durationSeconds": 0, "allowedFeatures": [] } { "success": true, "coupon": "STR" } Admin Only 관리자 권한 전용 신규 보안 라이선스 티켓 생성

서버는 verify-uid 요청을 수신하면 Supabase의 user_licenses 데이터 테이블에서 해당 고유 식별자의 만료 시각(expires_at)을 현재 시간 표준(Epoch Time)과 정밀 비교합니다. 만약 유효한 세션일 경우, 클라이언트 브라우저가 위변조할 수 없도록 암호화 서명된 클라이언트 전용 런타임 JWT 토큰을 함께 발행하여 클라이언트 내부 세션 컨텍스트로 반환하는 방식을 취합니다.

04. Phaser 3 CanvasPool 추적 및 하이재킹 알고리즘

PokeRogue 게임 클라이언트는 글로벌 범위에 명시적인 메인 씬(Scene) 인스턴스를 노출하지 않습니다. 따라서 본 mod.min.js 모듈이 게임 런타임 메모리를 제어하기 위해서는 브라우저 메모리 트리에 익명으로 등록된 Phaser 엔진의 코어 풀을 역추적해 들어가야 하는 기술적 과제가 존재합니다. 이를 위해 설계된 구조가 바로 CanvasPool 하이재킹 알고리즘입니다.

알고리즘은 창 내부의 글로벌 네임스페이스 window.Phaser의 존재 여부를 1차 검증한 후, 내부 구조를 다음과 같이 차례로 스캔합니다. 

function getBattleScene() {
  try {
    if (window.Phaser && window.Phaser.Display.Canvas.CanvasPool.pool.length > 0) {
      // 캔버스 엘리먼트를 점유하고 있는 최상위 부모 인스턴스에서 game 엔진 구조체 추출
      return window.Phaser.Display.Canvas.CanvasPool.pool[0].parent.game.scene.keys.battle;
    }
  } catch (e) {
    // 씬 스캔 에러 또는 런타임 로딩 미완료 시 즉시 트랩을 우회하여 크래시 방지
  }
  return null;
}

이 탐색 로직을 통해 확보된 battleScene 레퍼런스는 단순히 렌더링 화면만 바꾸는 것이 아니라, 게임 엔진이 매 웨이브마다 상태 데이터를 기록하는 핵심 스토리지인 gameData 및 현재 배틀 상태를 시뮬레이션하는 currentBattle 세그먼트에 대한 무제한적인 읽기/쓰기 권한을 가집니다. 모듈은 이 포인터를 메모리 버퍼에 캐싱하여 지속적으로 후킹 명령을 수행합니다.

05. 인게임 치트 모듈별 비즈니스 로직 세부 분석

대시보드가 사용자 UID 인증에 성공하여 라이선스를 획득하게 되면, 백그라운드 스케줄러를 통해 각 치트 옵션 기능의 플래그 상태(hackDataStore)를 모니터링하며 메모리 후킹을 속행합니다. 다음은 활성화되는 개별 모듈들의 심층 비즈니스 로직 및 메모리 필드 변조 기법의 상세한 분석입니다.

A. 아군 무적 모드 (God Mode)

플레이어의 포켓몬 파티 데이터 객체 배열인 battleScene.getParty()의 각 개체 요소에 접근하여, 데미지 산출 계산 메소드인 damage() 함수를 동적으로 인터셉트합니다. 기존 오리지널 데미지 함수를 복사 보관한 뒤, 이를 새로운 익명 프록시 함수로 대체시킵니다. 인게임 엔진이 플레이어 포켓몬에게 데미지를 인가하고자 damage(amount)를 트리거하면, 프록시가 매개변수 amount 값을 무조건 0으로 강제 삭감하여 전달하므로 어떠한 치명타나 고정 데미지에도 체력이 깎이지 않는 절대 무적 상태를 유지합니다.

B. 적군 즉사 모드 (One Hit Kill) 및 실드 무시 알고리즘

무적 모드와 정반대의 메커니즘이 적 필드 배열인 battleScene.getEnemyField()에 투사됩니다. 적 포켓몬의 damage() 함수가 호출되는 순간, 프록시 인터셉터는 대상을 즉사시키기 위해 amount 인자에 적의 현재 남아있는 최대 체력 값인 this.hp를 강제로 대입합니다. 더불어 보스 포켓몬이 웨이브 진행 도중 특정 체력 구간마다 발동시키는 하이퍼 실드 패턴인 '체력 세그먼트 스토퍼'를 우회하기 위해, 두 번째 파라미터인 세그먼트 무시 플래그(ignoreSegments) 값을 강제로 true로 오버라이딩하여 단 한 번의 공격으로 보스급 포켓몬까지 즉각 소멸시킵니다.

C. 야생 포켓몬 100% 이로치(Shiny) 강제 후킹 엔진

게임 내 인카운터 제너레이터가 작동하는 battleScene.currentBattle.enemyParty 배열 구조를 300ms 주기로 상시 감시합니다. 스케줄러는 필드에 등장한 야생 포켓몬 객체의 내부 플래그 필드인 .shiny 속성이 false 상태임을 감지하면 이를 true로 즉시 강제 갱신합니다. 동시에 이로치 고유 외형 등급인 .variant 필드 값에 0, 1, 2 사이의 난수를 임의 부여함으로써 상위 희귀 등급인 '레어 이로치(에픽)' 및 '럭셔리 이로치' 상태까지 실시간으로 로컬 메모리에 동적 바인딩시킵니다.

⚠️ 주의 사상 및 세이브 안정성

포획 확률 100% 모드(Always Catch) 활성화 시, 최종 스테이지(200층 등)의 무한다이노 바이옴 환경에서는 포획 명령이 내부 엔진 규칙에 의해 크래시를 유발할 수 있습니다. 본 모듈은 최종전 진입 감지 시 바이옴 상수(biomeType = 22)를 일반 평원 지대(0)로 속여 게임 아키텍처의 비정상 종료 예외를 안전하게 방어합니다.

06. 4종 바우처 동시 정밀 주입 알고리즘 정밀 해부

이전 세대 모듈의 가장 큰 문제점은 단일 정수 값만 제어하는 battleScene.gameData.vouchers 변수만을 수정했다는 점입니다. 이 경우 게임 화면의 텍스트 수치만 일시적으로 바뀔 뿐, 실제로 가챠(뽑기방) 씬으로 이동하여 바우처를 소모하려고 시도하면 서버 및 클라이언트 내부 상태의 불일치로 인해 0개로 강제 초기화되거나 갱신이 누락되는 심각한 바그 현상이 발생했습니다.

본 고도화 커널 모듈은 이를 해결하기 위해 Phaser UI 내부에서 참조하는 4종 하위 배열 구조체 동시 주입 및 렌더러 강제 리프레시 알고리즘을 구현하였습니다. 

function injectDetailedVouchers() {
  if (!isFeatureAllowed("money")) return alert("🚫 사용 권한이 부족합니다.");
  const battleScene = getBattleScene();
  if (!battleScene || !battleScene.gameData) return;

  const count = parseInt(document.getElementById("input-voucher-val").value) || 0;
  try {
    // 레벨 1: 기본 토탈 카운트 변수 동기화
    battleScene.gameData.vouchers = count;
    
    // 레벨 2: 뽑기방 코어 구조체인 4가지 바우처 등급별 객체 필드 정밀 오버라이드
    if (!battleScene.gameData.voucherCounts) {
      battleScene.gameData.voucherCounts = { "0": 0, "1": 0, "2": 0, "3": 0 };
    }
    for (let i = 0; i < 4; i++) {
      battleScene.gameData.voucherCounts[i] = count;
    }

    // 렌더러 리프레시: UI 핸들러를 물리적으로 가로채서 화면 갱신 함수를 강제 발동
    const handler = battleScene.ui?.getHandler();
    if (handler && typeof handler.updateVoucherCounts === "function") {
      handler.updateVoucherCounts();
    }

    // 세이브 파이프라인 연동: 변경 즉시 물리 세이브를 트리거하여 서버 로컬 스토리지 보존
    if (typeof battleScene.gameData.saveSystem === "function") {
      battleScene.gameData.saveSystem();
    }
    alert("🎫 [바우처 4종 통합 성공] 각 등급별 수량 동기화 완료!");
  } catch (e) {
    alert("정밀 바우처 변조 중 예외 발생: " + e.message);
  }
}

이 정밀 프로세스를 통해 일반 바우처, 대단한 바우처, 하이퍼 바우처, 마스터 바우처 등 인게임에 내장된 4가지 클래스의 등급 아이템이 완벽히 동일한 수치로 매핑되며 오작동 없이 안정적으로 가챠 씬에서 소모되도록 처리됩니다.

07. 스타터 및 도감 해금 데이터 직렬화 복구 시스템

도감 및 스타터 포켓몬 해금 기능(injectUnlockPityAndDex)은 게임 데이터베이스의 JSON 바이너리 직렬화 구조를 리버스 엔지니어링하여 모든 포켓몬의 수집 속성을 강제 개방합니다. 게임 데이터 세그먼트 내 dexData 객체와 starterData 객체는 사용자가 포켓몬을 조우하고 포획할 때마다 고유한 비트마스크 상수로 데이터를 저장합니다.

본 모듈은 이 비트마스크의 상한선을 메모리 단에서 임의로 오버플로우시켜 전체 도감을 완전 해금합니다.

  • seenAttr 및 caughtAttr 최대화: 자바스크립트가 안전하게 표현할 수 있는 최대 정수인 BigInt(Number.MAX_SAFE_INTEGER) 값을 비트 플래그에 강제 주입하여 전 포켓몬의 일반 폼, 이로치 폼 형상을 획득 상태로 인식시킵니다.
  • IVs 잠재력 오버클럭: 모든 개체의 개별 IV 배열 값([HP, 공격, 방어, 특공, 특방, 스피드])을 상한치인 31로 일괄 반복 대입 처리합니다.
  • 특성 및 알기술 마스터링: 숨겨진 특성 개방 속성인 abilityAttr = 7 값을 부여하고, 미해금 상태의 알기술 속성 테이블을 바인딩하여 스타터 선택창에서 모든 기술 세팅을 자유롭게 조율할 수 있도록 데이터 구조를 변혁합니다.

08. 백그라운드 후킹 가속 엔진 스케줄러 설계 사상

인게임 코어 치트 엔진(initCoreEngineHooks)은 단회성 실행 구조가 아닌, 브라우저가 살아있는 동안 상시 가동되는 정밀 스케줄러 인터벌 방식으로 설계되었습니다. 게임 프로세스는 매 웨이브가 종료되거나 새로운 전투 씬이 생성될 때마다 로컬 메모리의 변수 주소지를 초기화하고 새로운 포인터 객체들로 메모리를 대체하는 성향을 가집니다. 만약 치트 모듈이 최초 1회만 변조를 수행한다면, 다음 웨이브로 넘어가는 순간 모든 오버라이딩 상태가 증발하게 됩니다.

이를 방어하기 위해 본 모듈은 300ms(0.3초) 주기의 고속 폴링 엔진 루프를 가동합니다.

스케줄러 감시 타깃 필드 기동 주기 감출 매커니즘 및 후킹 동작 안전 조치 상태
battleScene.ui.handlers 300ms 상점 리롤 시 비용 연산 함수(getRerollCost)의 반환값을 무조건 0으로 가로챔 안전성 확보
StarterSelectUiHandler 300ms 스타터 포인트 제한 해제 함수(getValueLimit)를 오버라이드하여 999999999 코스트 반환 UI 크래시 방지 적용
battleScene.getEnemyParty() 300ms 포획 확률 난수 생성기 시드(randSeedInt)를 0으로 강제 락다운 수행 마스터볼 판정 우회

이 스케줄러 매커니즘 덕분에 게임 내부에서 포켓몬이 진화하거나, 파티 순서가 바뀌거나, 새로운 라운드가 생성되더라도 0.3초 이내에 프록시 함수가 재주입되어 사용자가 설정한 치트 상태가 게임 종료 시까지 영구 지속됩니다.

09. UI 인터페이스 정적 고정 및 드래그 보정 매커니즘

본 확장 프로그램의 대시보드 UI 패널(#mini-hack-container)은 게임 몰입감을 해치지 않으면서도 필요할 때 즉각 제어할 수 있도록 세밀한 CSS 고정 레이아웃과 마우스 좌표 변위 계산 기반의 물리 드래그 매커니즘이 적용되어 있습니다.

최초 로드 시 화면 정중앙이나 엉뚱한 레이아웃에 배치되어 게임 화면을 가리는 현상을 방어하기 위해 무조건 우측 하단의 정적 뷰포트 영역(bottom: 80px; right: 20px;)에 절대 좌표로 마운트되도록 제한을 걸었습니다. 이후 사용자가 패널의 상단 헤더 영역을 마우스로 클릭하여 드래그할 때 발생하는 마우스 모션 오프셋 변위 알고리즘은 다음과 같이 수행됩니다. 

// 드래그 시 마우스 현재 클라이언트 좌표와 UI 패널 좌상단 간의 델타 변위(Δ) 계산
let isDragMain = false;
let mx, my;

header.addEventListener("mousedown", (e) => {
  isDragMain = true;
  mx = e.clientX - container.getBoundingClientRect().left;
  my = e.clientY - container.getBoundingClientRect().top;
});

document.addEventListener("mousemove", (e) => {
  if (isDragMain) {
    // 계산된 변위를 기반으로 패널 좌표를 리얼타임 갱신
    container.style.left = `${e.clientX - mx}px`;
    container.style.top = `${e.clientY - my}px`;
    // 최초 지정되어 있던 bottom, right 속성을 무력화하여 좌표 꼬임 현상 완벽 방지
    container.style.bottom = "auto";
    container.style.right = "auto";
  }
});

document.addEventListener("mouseup", () => { isDragMain = false; });

이 드래그 보정 매커니즘을 통해 브라우저 창 크기가 조절되거나 해상도가 급격히 변하는 환경에서도 패널의 위치가 화면 밖으로 튕겨 나가지 않고 사용자가 드래그하여 배치한 커스텀 위치에 완벽하게 박제됩니다.

10. 장애 조치 지침 및 런타임 예외 처리 표준 선언

게임 클라이언트 버전 업데이트나 Phaser 엔진의 내부 난독화 명칭이 변경될 경우를 대비하여 본 커널은 고도화된 런타임 트랩 감지 및 예외 처리 아키텍처(Try-Catch Sandbox)를 상시 작동시킵니다. 개별 기능 구현 함수들은 내부가 전부 독립된 예외 블록으로 캡슐화되어 있어, 설령 특정 자금 변조 기능이 에러를 일으키더라도 그것이 메인 배틀 루프나 무적 모드 같은 다른 핵심 기능들의 가동 중단(Cascading Failure)으로 이어지지 않도록 구조적 절연 상태를 유지합니다.

주요 예외 대응 프로토콜 및 조치 가이드

  1. 서버 연결 시간 초과 (Render 504 Gateway Timeout): 웹소켓 및 REST API 응답이 5초 이상 지연될 경우, 대시보드는 로컬 스토리지에 백업된 캐시 세션을 탐색하여 마지막 인증에 성공했던 유효 라이선스 시간 범위 내에서 일시적인 오프라인 바이패스(Bypass) 모드를 작동시킵니다.
  2. activateAdvancedEggHack 및 이로치 참 함수 미정의 에러 해결: 구버전 소스에서 발생하던 스태브 누락 예외는 더미 프록시 인터페이스를 전면 배치함으로써 브라우저 콘솔 창에 Uncaught ReferenceError가 전사되는 현상을 완벽히 방어 완료했습니다.
  3. 클라이언트 메모리 누수 방지 조치: 300ms 단위로 기동되는 후킹 스케줄러의 부하를 줄이기 위해 이미 후킹 프록시 함수(_moddedGod 등)가 주입 완료된 개체 타깃들은 스케줄러 탐색 배열에서 즉시 제외시키는 조기 리턴(Early Return) 최적화 알고리즘이 할당되어 CPU 점유율을 1% 미만으로 방어합니다.
